## 0. TL;DR для быстрого въезда

- **Что есть:** одностраничный прототип на Claude Design (`Order_New_Dawn_dc.html`, ~7300 строк) — веб-клиент с полным визуалом боя (WebGL + 2D canvas), рабочей боевой моделью Starsector (flux / shield / armor / hull / overload / типы урона), редакторами (refit / forge / скиллы) и каркасом UI. Бой идёт против локального бота, **без сети и без сервера**.
- **Цель:** полноценный MMO-клиент к **авторитативному серверу на Go**. Большая галактика + real-time бои до 6 игроков, вся боевая модель Starsector.
- **Ключевое архитектурное решение:** сервер на Go — **источник истины**. Модель сети — **авторитативные снапшоты + клиентское предсказание** (НЕ детерминированный lockstep). Боевое ядро переписывается на Go с нуля (fixed timestep, мировые координаты, сидируемый RNG). Клиент переносит **только рендер и UI** из прототипа.
- **Первая веха:** сетевой вертикальный срез — 2 корабля (1 игрок + 1 серверный бот), один бой, WebSocket, снапшоты + предсказание своего корабля. НЕ начинать с 6 игроков и галактики.

## 1. Контекст и цель проекта

**Игра.** MMO в духе Starsector: большая галактика, real-time космические бои до 6 игроков, полная боевая модель (щиты с дугой и резистами, flux, armor, hull, overload, множество орудий с разными типами урона, кастомные сетапы кораблей).

**Роль прототипа.** Прототип создавался, чтобы (а) доказать, что весь нужный визуал реализуем на вебе (лазеры, снаряды, ракеты, взрывы, преломление пространства), и (б) сформировать каркас интерфейса с редакторами. Обе задачи выполнены. Прототип — **не фундамент боевой логики**, а источник визуала и UI.

**Команда.** Один человек + Claude как программист-напарник. Важные следствия:
- Claude не видит рантайм — визуал и «фил» оцениваешь ты, Claude даёт код.
- Claude забывает контекст между сессиями — отсюда этот документ и требование вести его в актуальном состоянии.
- Claude уверенно ошибается в деталях API конкретных версий — весь код проверяется запуском, не принимается на веру.
- Claude силён в текстовом коде (Go, TypeScript, GLSL) — стек выбран так, чтобы это использовать по максимуму.

## 2. Целевая архитектура

### 2.1. Два слоя (не смешивать)

```
СЛОЙ ГАЛАКТИКИ / ПЕРСИСТЕНТНОСТИ          СЛОЙ БОЯ (battle instance)
- аккаунты, инвентарь кораблей            - авторитативная симуляция боя
- карта галактики, флоты, экономика       - fixed timestep 30–60 Гц
- матчмейкинг                             - до 6 игроков, полная модель Starsector
- тики в секундах, БД                     - goroutine на бой, эфемерное состояние
                  \                       /
                   \  бой заканчивается → результат возвращается в слой галактики
```

Каждый бой — отдельная goroutine (дёшево по памяти, легко масштабируется горизонтально). Один Go-бинарник может хостить оба слоя на старте; позже разносится на отдельные сервисы.

### 2.2. Раздел ответственности клиент ↔ сервер

```
┌─ Go server (АВТОРИТЕТ) ───────────────┐        ┌─ TS client (глаза и руки) ──────────────┐
│ • fixed timestep (напр. 30–60 Гц)     │        │ • весь рендер боя (WebGL/2D) из прототипа│
│ • МИРОВЫЕ координаты (юниты, не px)    │        │ • UI + редакторы (refit / forge / скиллы)│
│ • flux / shield / armor / hull /      │◄──────►│ • интерполяция чужих кораблей            │
│   overload / hit-resolve / урон       │ WS/WT  │ • prediction + reconciliation своего     │
│ • сидируемый RNG (только сервер)      │        │ • локальные снаряды/партиклы/варпы       │
│ • валидация лоадаутов                  │        │ • ввод → инпуты на сервер               │
└───────────────────────────────────────┘        └──────────────────────────────────────────┘
```

**Правило тонкого канала.** Сервер шлёт авторитативное **состояние кораблей** (позиция, скорость, угол, hull, flux, shield, overload, статусы) + **события** («орудие X выстрелило в тик T», «взрыв тут», «overload у Y», «корабль убит»). Полёт снарядов, партиклы, heat-марки, преломление пространства клиент **проигрывает локально** как косметику поверх авторитативного состояния. Хит и урон считает **только сервер**. Это держит трафик тонким даже при сотнях снарядов и позволяет перенести рендер снарядов из прототипа почти без изменений.

### 2.3. Сетевая модель (снапшоты + предсказание)

Классическая модель (Valve/Overwatch-подобная):
- Сервер тикает физику на 30–60 Гц, шлёт снапшоты на 20–30 Гц.
- Клиент рендерит на 60/144 Гц, **интерполируя между двумя последними снапшотами** с буфером ~100 мс.
- **Свой** корабль — предсказание + сверка (reconciliation) для отзывчивого управления.
- **Чужие** корабли — интерполяция.

**Почему НЕ lockstep:** детерминированный lockstep потребовал бы побитово идентичной симуляции на Go и на TS одновременно — неоправданно дорого при разноязычном стеке. Снапшотная модель — правильный выбор здесь.

### 2.4. Транспорт и сериализация

- **Транспорт (старт): WebSocket.** Зрелые Go-библиотеки, заводится быстро, для 6 игроков со снапшотами ~20–30 Гц вывезет.
- **Транспорт (позже): WebTransport (QUIC, UDP-подобный).** Лучше для real-time — у WebSocket поверх TCP есть head-of-line blocking (одна потеря пакета → лаг-спайк). **Заложить слой сообщений транспорт-агностично**, чтобы перейти без переписывания. ⚠️ Зрелость WebTransport в браузерах и в Go проверять на момент реализации — статус быстро меняется, не полагаться на память.
- **Сериализация: protobuf** (кодоген для Go и TS). Не гонять JSON каждый тик. Позже — дельта-кодирование снапшотов (слать только изменившиеся поля).

### 2.5. Единый источник боевой модели (КРИТИЧНО)

Статы оружия, кораблей, резисты щитов, стоимость флукса, порог overload и т.д. — в **общей схеме данных** (JSON/protobuf), которую грузят и Go, и TS. **Никогда не дублировать числа руками в двух местах.** Иначе баланс клиента и сервера разъедется → «на клиенте попал, на сервере нет». Авторитетную логplanику урона держит Go; клиент держит тонкое *предсказанное зеркало* той же логики для мгновенного фидбека.

## 4. Критические подводные камни, найденные в прототипе

Эти три вещи в прототипе делают симуляцию несетевой. **Они НЕ чинятся в JS — соответствующий код не переносится, а переписывается на Go сразу правильно.** Но знать их надо, чтобы не скопировать проблему.

1. **Переменный `dt`** (строка ~2521): `dt = Math.min(2.5, (ts - lastTs) / 16.67)` — шаг привязан к реальному времени кадра. На Go нужен **фиксированный шаг** + накопитель времени; рендер интерполирует между тиками.

2. **Мир в пикселях экрана** (~173 обращения к `cssH/cssW/DPR`, в т.ч. внутри физики). Пример (строка ~3001): `shieldR = s.radius * 1.7 / cssH`. Радиусы, дистанции, позиции — в координатах, производных от размера канваса. Headless-сервер канваса не имеет. На Go — **разрешение-независимые мировые координаты (юниты)**; `cssH/DPR` живут строго в слое рендера клиента.

3. **`Math.random()` в игровой логике** (~60 вызовов: PD-роллы, разброс попаданий). На Go — **сидируемый PRNG, только на сервере**. Клиент рандом в логике не использует (косметический рандом партиклов — можно, он не влияет на состояние).

4. **Зависимость от `x-dc` / `support.js`** — это шаблонизатор прототипа Claude Design, в проде его не будет. UI переносится на Vite+TS.

## 7. Приложение B — Рендер-модуль `battle-gl.js` и sim→render контракт

> Найдено при анализе `battle-gl.js` (~992 строки). Это ключ к Фазе 1: рендер уже отделён от симуляции через объект `B` и уже работает в мировых координатах. Go-снапшот маппится на этот объект.

### Публичный API
```js
const r = BattleGL.create(canvas);   // null если WebGL недоступен
r.render(B);                          // отрисовать кадр из объекта симуляции B
r.resize();                           // подхватить размер канваса
r.toScreen(B, x, y) -> [sx, sy];      // мир→экран последнего кадра (для оверлеев/турелей)
r.worldScale(B) -> number;            // юниты→пиксели
r.dispose();
```

### Проекция камеры (УЖЕ разрешение-независима)
```
worldScale(B) = B.zoom * Hcss           // Hcss = высота канваса в css-px
sx(B,x) = (x - B.camX) * worldScale + Wcss*0.5
sy(B,y) = (y - B.camY) * worldScale + Hcss*0.5
```
Вывод: `x/y` кораблей — **мировые координаты**; размер экрана входит только здесь, на границе проекции. Камера (`camX/camY/zoom`) — **клиентская**, с сервера НЕ приходит. Мир нормирован по высоте — при портировании на Go задать явные мировые юниты и убрать зависимость от `Hcss` (она остаётся только в клиентском рендере).

### Объект `B` (то, что рендер читает каждый кадр)
- `B.ships` — массив кораблей (см. ниже).
- `B.projs` — снаряды. **Клиентские/предсказанные**, не из снапшота целиком.
- `B.parts` — партиклы (искры/обломки). Чисто косметика, клиент.
- `B.warps` — ударные волны рефракции пространства от взрывов. Клиент.
- `B.asteroids` — статичные препятствия сцены.
- `B.camX`, `B.camY`, `B.zoom` — камера (клиент).
- `B.t` — время сцены (для анимаций шейдеров).
- `B.gl` — WebGL-контекст (внутреннее).

### Поля корабля `s`, читаемые рендером (контракт снапшота)
**Из авторитативного снапшота сервера (мировое состояние):**
`x`, `y`, `angle`, `radius`, `side`, `dead`, `hitT` (вспышка урона), `hullTex` (какой корпус), `hullAspect`.

**Состояние щита (сервер):**
`shieldA` (угол дуги), `shieldArc`, `shieldDeploy` (0..1 развёрнутость), `shieldBand`, `shieldCol`, `shieldFlicker`, `sHits` (точки попаданий по щиту — для ряби).

**Состояние луча/стрельбы (сервер шлёт как событие/флаг, клиент проигрывает):**
`beam` (интенсивность 0..1), `beamTgt`, `beamTX`, `beamTY`, `beamMuzWX`, `beamMuzWY`, `beamCore`, `beamCoreOuter`, `beamCoreW`, `beamGlow`, `beamGlowOuter`, `beamGlowW`, `beamImpact`.

**Косметика/материалы (клиент, из пресета корабля):**
`metal`, `rough`, `tint`, `heat` (heat-марки), `dieGlow`, `vanish` (fade при смерти), `length`, `width`.

> ⚠️ Разделение при проектировании снапшота: авторитетно и по сети идут `x/y/angle/hull/flux/shield*/overload/dead`. Лучи и попадания — как **события** (`fire`, `beamOn/off`, `shieldHit`), клиент из них восстанавливает `beam*`/`sHits`. Материалы (`metal/rough/tint/hullTex`) клиент берёт из пресета по id корабля — по сети каждый кадр НЕ гонять.

## 8. Приложение C — Глоссарий боевой модели

- **flux** — «тепло/заряд», копится от стрельбы своими орудиями и от урона по щиту; рассеивается (dissipation) со временем. `flux ≥ fluxMax` → **overload**.
- **overload** — при максимальном flux корабль теряет щит и системы (в прототипе ~10с), дрейфует по инерции, не стреляет.
- **shield** — дуга перед кораблём (`shieldArc`), можно поднять/опустить, поглощает урон переводя его во flux; имеет резисты по типам урона.
- **armor / hull** — прочность корпуса; в прототипе hull-модель есть, полноценная сетка брони (armor grid) — цель будущей проработки на Go.
- **типы урона** — kinetic, HE, energy, bio; по-разному взаимодействуют со щитом/hull (в прототипе HE: ×0.7 во flux по щиту, ×1.5 по hull).
- **damage type resists** — на щите: `resKinetic/resHE/resEnergy/resBio` (%).
- **subsystem damage** — попадания ракет могут крутить тягу (engine hit) или глушить огонь (weapon hit).
